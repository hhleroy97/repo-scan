# Progressive Disclosure

## Summary

This Nielsen Norman Group article defines progressive disclosure as a UX pattern that shows only the most important options initially and reveals specialized features on request, balancing user demand for power with the need for simplicity. It covers design criteria, research-backed benefits (learnability, efficiency, fewer errors), and contrasts progressive disclosure with staged disclosure (e.g., wizards), including when each approach fits task structure.

## Key claims

- Progressive disclosure satisfies conflicting user needs by deferring secondary features until explicitly requested, so most users complete tasks without added complexity.
- The initial display signals importance: what appears first is what users should prioritize, helping novices avoid mistakes and experts skip rarely used options.
- Effective progressive disclosure requires two things: the right split between primary and secondary features, and an obvious, well-labeled path to advance with strong information scent.
- Prioritize primary features using task analysis, field studies, frequency-of-use statistics, analytics, and observational usability testing—not analytics alone.
- Designs with more than two disclosure levels usually hurt usability; if complexity demands it, chunk advanced options into sensible groups and test with real tasks.
- Multiple secondary displays or multiple paths to secondary options add initial-level complexity and should be used sparingly, with clear labeling.
- Staged disclosure (linear step-by-step, e.g., wizards) differs from progressive disclosure (hierarchical, optional deeper levels): staged suits independent task steps; progressive suits optional advanced settings.
- Task interdependence matters: features used together with frequent back-and-forth belong on one screen; features needed only in later phases (e.g., payment after exploration) should be deferred.
